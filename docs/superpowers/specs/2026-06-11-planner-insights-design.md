# 기획자용 인사이트 대시보드 확장 설계

작성일: 2026-06-11

## 목적

현재 대시보드는 "무엇을 얼마나 썼나"(현황)까지만 답한다. 개인 사용자이면서 직업이 **기획자**인 사용자가 "내가 Claude로 **어떤 일**을 **얼마에** 했고, **잘** 쓰고 있나"를 판단할 수 있도록 효율·변화 관점을 추가한다.

### 오디언스와 그에 따른 방향
- **오디언스**: 개인 사용자 / 직업은 기획자 (코드 직접 작성보다 기획·문서·분석 용도)
- 기기·팀원 비교, 조직 ROI 보고는 범위 외(개인용).
- 모델 선택(Opus/Sonnet)은 기획자가 직접 바꾸는 레버가 아니므로 **비용 투명성** 수준으로만 노출.
- "무슨 일에 썼나"(작업유형)를 핵심 관심사로 강화.
- 모든 신규 지표는 기존과 동일하게 **평이한 언어**로 표기(토큰→분량, 캐시→재활용/아낀 비용 등).

## 비범위 (Non-goals)
- 에러·재시도·작업 성공률: 업로더가 `assistant`의 `tool_use`만 파싱하므로 데이터 보강이 선행돼야 함 → 이번 범위 제외.
- 드릴다운(차트 클릭→상세) 인터랙션: 단일 HTML 상태관리 복잡도 대비 개인용 효용 낮음 → 제외.
- 스키마(`supabase_schema.sql`)·업로더(`claude_usage_uploader.py`) 수정 없음. **전적으로 기존 수집 데이터만 사용.**

## 전제 / 기존 구조
- 단일 파일 `dashboard.html` (서버 불필요), Supabase 직접 조회, Chart.js v4.
- 토큰·비용·캐시 집계는 `(session_id, timestamp)` 단위 dedupe 규칙을 따른다(한 메시지의 여러 tool_use가 동일 토큰을 행마다 복제하므로 메시지당 1회만 집계). **신규 지표도 이 규칙을 그대로 따른다.**
- 비용 단가표(기존): Opus $5/$25, Sonnet $3/$15, Haiku $1/$5 per 1M; 캐시 쓰기 ×1.25, 읽기 ×0.1. 모델 미상은 Opus 단가로 보수 추정.
- 기간 필터(전체/일/주/월)·기기 필터·라이트/다크 모드는 신규 블록에도 동일 적용.

## 추가 구성 (4개 블록)

### ① 작업(세션) 단위 뷰 — 신규
"이 작업 하나에 얼마 들었지?"에 답한다.

- **요약 카드 2개**
  - `작업 1건당 평균 비용`: 기간 내 고유 `session_id` 수로 총비용을 나눈 값(USD).
  - `가장 비쌌던 작업`: 최고 비용 세션의 비용(USD). 카드 설명에 해당 프로젝트명 표기.
- **표: 비싼 작업 Top 10**
  - 컬럼: 날짜(세션 첫 timestamp) · 프로젝트 · 비용(USD) · 주고받은 분량(입력+출력 토큰) · 도구 호출 수.
  - 집계 단위: `session_id`. 비용/분량은 dedupe 규칙으로, 도구 호출 수는 행 수(dedupe 전)로 센다.
  - 정렬: 비용 내림차순, 상위 10개.

### ② 작업유형 심화 — 기존 도넛 확장
기존 `chart-worktype` 도넛(작업유형 분포)은 유지하고, 아래 2개를 추가한다.

- **작업유형별 비용** (수평 바): 코드편집·탐색·실행·문서 등 분류별 비용 합(USD).
  - 작업유형 분류는 기존 도넛이 쓰는 분류 로직을 재사용한다(중복 정의 금지).
- **작업유형 추세** (스택 영역/바): 기간 버킷별로 작업유형 구성 변화. "이번 달엔 탐색보다 작성에 더 썼다"가 보이게.

### ③ 작업 리듬 히트맵 — 신규
"나는 언제 집중적으로 쓰나"를 본다.

- 요일(7행) × 시간대(24열) 격자, 셀 색 진하기 = 해당 칸의 작업 실행 횟수(또는 비용 중 택1, 기본: 작업 실행 횟수).
- `timestamp`의 로컬 요일/시각 기준(`getDay`/`getHours`).
- Chart.js에 히트맵 기본 타입이 없으므로 **경량 DOM 격자**(CSS grid + 배경색 보간)로 구현. 라이트/다크 모드 색 토큰을 따른다.

### ④ 변화·절감 요약 — 신규 요약 카드
절대값만 보던 화면에 변화 신호를 더한다.

- `지난주 대비 비용`: 직전 동일 기간 대비 증감 %와 방향(▲/▼). 기간 필터가 '주'가 아니어도 항상 직전 7일 vs 그 전 7일 기준으로 계산(고정 기준이라 해석이 일관됨).
- `아낀 비용(재활용)`: 캐시 읽기 토큰을 정상 입력가로 환산했을 때와 캐시 읽기가(×0.1)의 차액 = 재활용으로 절감한 추정액(USD).
- **보조 카드(작게)**: `비싼 모델 비중` — 전체 비용 중 Opus(또는 미상→Opus 추정) 비용 비율 %. 비용 투명성 목적의 참고 지표.

## 데이터 흐름
1. 기존대로 Supabase `tool_calls` 전량 로드 → `allData`.
2. 기간/기기 필터 적용된 행 집합에서:
   - dedupe(`session_id|timestamp`) 집합 → 비용/분량/캐시 계열 (①④②, 카드 전반).
   - 비-dedupe 행 → 도구 호출 수(① 표), 작업유형 분류(②), 히트맵 횟수(③).
3. `renderAll()`에 신규 렌더 함수 추가, `destroyCharts()`에 신규 Chart 인스턴스 등록(히트맵 DOM은 별도 비우기).

## 레이아웃 배치
- 요약 카드 영역(`.summary`): ④의 카드 3개(`지난주 대비`·`아낀 비용`·`비싼 모델 비중`)와 ①의 카드 2개(`작업 1건당 평균 비용`·`가장 비쌌던 작업`)를 기존 카드 뒤에 추가.
- 차트 영역(`#charts-tpl`): 상단 근처에 ① 표 패널 → ② 작업유형 비용/추세 패널(기존 worktype 도넛 옆) → ③ 히트맵 패널(전폭) 순으로 삽입.

## 오류 처리
- 데이터 0건/세션 0개: 카드는 `-`, 표·히트맵은 "데이터 없음" 안내(기존 빈 상태 처리 방식 따름).
- 직전 기간 비용 0: `지난주 대비`는 증감% 대신 "비교 데이터 없음" 표기(0으로 나누기 방지).
- 모델 미상 행: 기존 규칙대로 Opus 단가 추정, `비싼 모델 비중`에는 Opus로 합산.

## 테스트 / 검증
- 단일 HTML이라 자동화 테스트 없음. 검증은 브라우저 수동 확인:
  - 기간/기기 필터 전환 시 신규 블록이 함께 갱신되는지.
  - 라이트/다크 토글 시 히트맵 색이 테마를 따르는지.
  - dedupe 적용 계열(비용)과 비적용 계열(도구 호출 수)이 의도대로 다른지(같은 세션의 멀티 tool_use 케이스로 확인).
  - 빈 데이터·직전기간 0 비용에서 깨지지 않는지.
